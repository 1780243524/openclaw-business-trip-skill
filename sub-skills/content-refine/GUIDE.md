# Content Refine — 内容微调与偏好学习

## 职责

1. **精准微调** — 对已生成文档的局部或全局内容进行修改
2. **提炼规则** — 从每次修改中抽取可复用的偏好规则
3. **持久学习** — 将规则写入偏好库，影响未来所有内容生成

---

## 偏好库位置

偏好库存储于 skill 根目录下的 `preferences/` 目录。

**执行前先确定 SKILL_DIR：**
```bash
SKILL_DIR=$(dirname "$(find ~/.openclaw/workspace/skills /projects/.openclaw/skills -name 'SKILL.md' -path '*/business-trip-assistant/SKILL.md' 2>/dev/null | head -1)")
```

```
$SKILL_DIR/preferences/
  writing-style.md    ← 写作风格（语气、详略、人称、段落格式）
  content-rules.md    ← 内容规则（特定替换、结构要求、禁用词等）
  corrections-log.md  ← 历史修正记录（带时间戳的原文→修改后+提炼规则）
```

**若目录/文件不存在，执行时自动创建并初始化。**

## 工作流程

### Step 0 — 读取偏好库上下文
执行任何操作前，先读取：
- `preferences/writing-style.md`
- `preferences/content-rules.md`

了解当前已有的偏好规则，避免重复添加或互相矛盾。

---

### Step 1 — 识别修改意图

根据用户输入判断场景：

| 场景 | 示例 | 处理方式 |
|------|------|---------|
| A. 直接告知修改内容 | "第二段改成：…" | 立即执行修改 |
| B. 描述问题 | "这里太啰嗦了" | 理解意图后提出修改方案，确认后执行 |
| C. 声明偏好规则 | "以后报告里不要用'综上所述'" | 直接写入偏好库，无需修改现有文档 |
| D. 混合 | "这段改一下，以后也要这样" | 执行修改 + 同步写入偏好库 |

---

### Step 2 — 定位目标文档

若用户未指明文档，按优先级查找：
1. 当前对话中最近提到的文档
2. `data/` 目录下最近修改的 trip 对应的最新文档（report.md → meeting-001.md → schedule.md）
3. 若仍无法确认，询问用户："请问您要修改的是哪个文档？"

---

### Step 3 — 执行修改

**局部修改：**
- 精确定位要修改的段落/句子
- 仅替换目标内容，保持其余内容和格式不变
- 使用 `edit` 工具进行 oldText/newText 替换

**全局调整：**
- 列出将要修改的所有位置（≤10条逐一列出，>10条先汇总）
- 确认后批量执行

**同步 .docx：**
- 若对应 `data/{trip-id}/` 目录中存在 `.docx` 文件，同步更新 .docx（使用 OOXML 替换或重新生成）
- 同步更新企微文档（调用 wecom-doc skill）

---

### Step 4 — 提炼并存储偏好规则

**每次修改后必须执行此步骤。**

#### 提炼原则
从修改行为中抽取**可泛化的规则**，而非记录具体内容。

示例：
- 用户把"综上所述，本次会议…" → "本次会议…" → 规则：`报告结尾不使用"综上所述"等套话开头`
- 用户把某段3行压缩成1行 → 规则：`会议纪要每个议题描述控制在2句以内`
- 用户把"负责人：XXX" → "owner：XXX" → 规则：`英文术语用英文标注，不翻译（如owner、deadline、action item）`

#### 写入格式

追加到 `preferences/corrections-log.md`：
```markdown
## [{YYYY-MM-DD HH:mm}] 修正记录

**文档：** {trip-id} - {文档类型}
**原文：** `{原始内容摘要}`
**修改为：** `{修改后内容摘要}`
**提炼规则：** {规则描述}
**已同步到：** {writing-style.md / content-rules.md / 未泛化}
```

若规则可泛化，追加到对应偏好文件：
- **写作风格类** → `writing-style.md`（语气、详略、人称、段落结构）
- **内容规则类** → `content-rules.md`（替换规则、禁用词、结构要求）

---

### Step 5 — 告知用户

修改完成后，输出简洁摘要：
```
✅ 已修改：{修改点简述}

📌 新增偏好规则：
  · {规则1}
  · {规则2}（如有）

下次生成{文档类型}时将自动应用上述规则。
```

## 被其他子 skill 调用的规则注入

其他子 skill（meeting-notes、schedule-gen、report-gen）在生成内容前，必须：

1. 检查 `preferences/writing-style.md` 和 `preferences/content-rules.md` 是否存在且非空
2. 若存在有效规则，将规则内容作为**系统约束**注入到生成 prompt 中
3. 在输出内容时，附注："📌 已应用 {N} 条个人偏好规则"

**检查方式：**
```
$SKILL_DIR/preferences/writing-style.md
$SKILL_DIR/preferences/content-rules.md
```

## 特殊指令

| 用户说 | 行为 |
|--------|------|
| "查看我的偏好规则" | 读取并展示两个偏好文件的现有规则 |
| "清除偏好规则" | 确认后清空对应文件（保留文件，清空规则内容） |
| "删除某条规则" | 定位并删除指定规则，更新文件 |
| "只记住，不修改文档" | 仅写偏好库，不改动任何文档 |
| "只改这次，不记住" | 执行修改，不写偏好库 |

---

## 注意事项

- **不臆测规则**：规则只从用户的实际修改行为中提炼，不主动推断用户"可能"的偏好
- **规则冲突检测**：写入前检查是否与现有规则矛盾，若有冲突，提示用户选择保留哪条
- **规则精简**：同类规则合并，避免偏好库膨胀影响效果
- **隐私保护**：偏好库记录规则和修改摘要，不存储完整的会议内容或敏感信息
